3. What are differences between HTTP version 1.0 and 1.1?


3.1. List of differences between 1.0 and 1.1

Підтримка проксі та поле Host:
HTTP 1.1 має обов'язковий заголовок Host за специфікацією.
HTTP 1.0 офіційно не вимагає заголовку Host, але це не зашкодить додати один, багато програм (проксі) очікують побачити заголовок Host незалежно від версії протоколу.

Постійні зв'язки:
HTTP 1.1 також дозволяє мати постійні підключення, що означає, що у вас може бути більше одного запиту/відповіді на одному HTTP-з'єднанні.
У HTTP 1.0 вам довелося б відкрити нове підключення для кожної пари запиту/відповіді. І після кожної відповіді зв'язок буде закритий. Це призводить до деяких проблем із високоюефективністю через TCP Slow Start.
Метод OPTIONS:
HTTP / 1.1 вводить метод OPTIONS. HTTP-клієнт може використовувати цей метод для визначення можливостей HTTP-сервера. В основному це використовується для спільного використання ресурсів Cross Origin у веб-програмах.
Кешування:
HTTP 1.0 підтримує кешування за допомогою заголовка: If-Modified-Since.
HTTP 1.1 розширює підтримку кешування багато чого, використовуючи те, що називається тегом entity. Якщо 2 ресурси однакові, то вони матимуть ті самі теги об'єктів. HTTP 1.1 також додає умовні заголовки If-Unmodified-Since, If-Match, If-None-Match. Є також додаткові доповнення щодо кешування, як заголовок Cache-Control.

3.2. Answer to question - how to pass user-data via GET? Via POST? What is the diff?

Основна відмінність методів GET і POST - в способі передачі даних веб-форми обробленому скрипту, а саме:

Метод GET відправляє скрипту всю зібрану інформацію форми як частина URL:

http://www.komtet.ru/script.php?login=admin&name=komtet


Метод POST передає дані таким чином, що користувач сайту вже не бачить передаються скрипту дані:

http://www.komtet.ru/script.php


Обидва методи успішно передають необхідну інформацію з веб-форми скрипту, тому при виборі того чи іншого методу, який буде найбільш підходити сайту, потрібно враховувати наступні фактори:
  1. Принцип роботи методу GET обмежує обсяг переданої скрипту інформації;
  2. Оскільки метод GET відправляє скрипту всю зібрану інформацію форми як частина URL (тобто у відкритому вигляді), то це може згубно вплинути на безпеку сайту;
  3. Сторінку, що згенерувала методом GET, можна помітити закладкою (адреса сторінки буде завжди унікальний), а сторінку, що згенерувала метод POST не можна (адреса сторінки залишається незмінним, так як дані в URL-адреса не підставляються);
  4. Використовуючи метод GET можна передавати дані не через веб-форму, а через URL сторінки, ввівши необхідні значення через знак &:

    http://www.komtet.ru/script.php?login=admin&name=komtet

  5. Метод POST на відміну від методу GET дозволяє передавати запитом файли;
  6. При використанні методу GET існує ризик того, що пошуковий робот може виконати той чи інший відкритий запит.

6. Using the hurl.it


Connection - статус з`єднання;
Content-Encoding - алгоритм стиснення;
Content-Security-Policy-Report-Only - дозволяє експериментувати з політиками шляхом моніторингу (але не дотримання) їхніх наслідків;
Content-Type - тип носія ресурсу;
Date - дата і час створення повідомлення;
Server - інформація про програмне забезпечення, яке використовується hurl.it для обробки запиту
Strict-transport-security - дозволяє веб-сайту повідомити браузеру, що він повинен бути доступний тільки через HTTPS, замість того, щоб використовувати HTTP.
Transfer-Encoding - визначає форму кодування, що використовується для передачі безпечного входу користувача.

8. What is the different with GET request?


При HEAD-request сервер не повертає вміст http-відповіді, лише пусте тіло документу, є код відповіді і HTTP headers. Застосовується для отримання метаданих, перевірки наявності ресурсу (валідації URL) і щоб дізнатися, чи не змінився він з моменту останнього звернення
